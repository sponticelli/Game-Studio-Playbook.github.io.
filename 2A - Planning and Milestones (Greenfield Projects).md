# 2A.1 Milestone Architecture

## Why It Matters

Without clear milestones, "progress" becomes subjective. Teams work hard but can't answer "are we on track?" Milestones create shared checkpoints where everyone agrees on what "done" looks like. They're also decision points—moments to evaluate whether to continue, pivot, or kill.

## Framework: Standard Milestone Progression

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  PROTOTYPE  │───▶│  VERTICAL   │───▶│   ALPHA     │───▶│   BETA      │
│             │    │   SLICE     │    │             │    │             │
│ "Can this   │    │ "Is this    │    │ "Does it    │    │ "Is it      │
│  work?"     │    │  fun?"      │    │  all work?" │    │  stable?"   │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
						                                                            │
						                                                            ▼
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│    LIVE     │◀───│   GLOBAL    │◀───│   SOFT      │◀───│  RELEASE    │
│             │    │   LAUNCH    │    │   LAUNCH    │    │  CANDIDATE  │
│ "Grow &     │    │ "Ship to    │    │ "Do metrics │    │ "Ready to   │
│  sustain"   │    │  everyone"  │    │  work?"     │    │  ship?"     │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

## Framework: Milestone Definition Template

### Milestone: PROTOTYPE

|Attribute|Definition|
|---|---|
|**Purpose**|Prove core mechanic is technically feasible and potentially fun|
|**Duration**|2-8 weeks|
|**Team Size**|1-3 people|
|**Quality Bar**|Placeholder art acceptable, bugs acceptable, PC/dev build only|
|**Key Question**|"Can this mechanic work?"|

**Entry Criteria:**

- [ ] Core loop defined
- [ ] Technical approach identified
- [ ] Team assigned

**Exit Criteria:**

- [ ] Core mechanic playable
- [ ] Technical feasibility confirmed
- [ ] Initial fun signal (or decision to pivot)
- [ ] Go/no-go decision made

---

### Milestone: VERTICAL SLICE

|Attribute|Definition|
|---|---|
|**Purpose**|Prove the game is fun with representative content and quality|
|**Duration**|1-3 months|
|**Team Size**|Core team|
|**Quality Bar**|Near-final quality for included content, representative of target experience|
|**Key Question**|"Is this fun? Would someone want to play this?"|

**Entry Criteria:**

- [ ] Prototype validated
- [ ] Art direction defined
- [ ] Scope document approved

**Exit Criteria:**

- [ ] 10-30 minutes of polished, representative gameplay
- [ ] Core loop complete with progression hooks visible
- [ ] Art at target quality level
- [ ] External playtest feedback positive
- [ ] Go/no-go decision made

---

### Milestone: ALPHA

|Attribute|Definition|
|---|---|
|**Purpose**|All features implemented, content in progress|
|**Duration**|2-6 months|
|**Team Size**|Full team|
|**Quality Bar**|Feature complete, content may be placeholder, bugs expected|
|**Key Question**|"Does everything work together?"|

**Entry Criteria:**

- [ ] Vertical slice approved
- [ ] Full feature list locked
- [ ] Production pipeline established

**Exit Criteria:**

- [ ] All planned features functional
- [ ] 50%+ content implemented
- [ ] No critical blockers
- [ ] Monetization systems functional (if F2P)
- [ ] Analytics tracking verified

---

### Milestone: BETA

|Attribute|Definition|
|---|---|
|**Purpose**|Feature frozen, content complete, bug fixing and polish|
|**Duration**|1-3 months|
|**Team Size**|Full team, QA ramped up|
|**Quality Bar**|Content complete, polish pass done, only bug fixes|
|**Key Question**|"Is this stable and polished enough to ship?"|

**Entry Criteria:**

- [ ] Alpha complete
- [ ] Feature freeze in effect
- [ ] All content implemented

**Exit Criteria:**

- [ ] All content final
- [ ] No critical or major bugs
- [ ] Performance targets met
- [ ] Localization complete
- [ ] Platform certification requirements met

---

### Milestone: RELEASE CANDIDATE

|Attribute|Definition|
|---|---|
|**Purpose**|Final validation before soft launch or release|
|**Duration**|1-4 weeks|
|**Team Size**|QA focus, limited dev|
|**Quality Bar**|Shippable—if no issues found, this is the build|
|**Key Question**|"Can we ship this build?"|

**Entry Criteria:**

- [ ] Beta complete
- [ ] All known bugs triaged
- [ ] Platform submission requirements met

**Exit Criteria:**

- [ ] Passes full regression
- [ ] Stakeholder sign-off
- [ ] Store assets approved
- [ ] Launch plan finalized

---

### Milestone: SOFT LAUNCH (Mobile/F2P)

|Attribute|Definition|
|---|---|
|**Purpose**|Test with real users in limited markets, validate metrics|
|**Duration**|4-16 weeks|
|**Team Size**|Full team on standby for iteration|
|**Quality Bar**|Production quality, limited content acceptable|
|**Key Question**|"Do the metrics support global launch?"|

**Entry Criteria:**

- [ ] Release candidate approved
- [ ] Analytics fully instrumented
- [ ] Soft launch markets selected
- [ ] Success criteria defined

**Exit Criteria:**

- [ ] Retention targets met (or decision to iterate/kill)
- [ ] Monetization targets met
- [ ] Crash rate acceptable
- [ ] Critical issues resolved
- [ ] Go/no-go for global launch

---

### Milestone: GLOBAL LAUNCH

|Attribute|Definition|
|---|---|
|**Purpose**|Full release to all target markets|
|**Duration**|Event (with 2-4 week launch window)|
|**Team Size**|All hands on deck|
|**Quality Bar**|Production|
|**Key Question**|"Is the launch successful?"|

**Entry Criteria:**

- [ ] Soft launch success (or waived for premium)
- [ ] Marketing ready
- [ ] Support ready
- [ ] Live ops calendar ready

**Exit Criteria:**

- [ ] Successful launch week
- [ ] No critical incidents
- [ ] Transition to live ops complete

---

### Milestone: LIVE (Ongoing)

|Attribute|Definition|
|---|---|
|**Purpose**|Sustain and grow the game post-launch|
|**Duration**|Ongoing|
|**Team Size**|Sustain team|
|**Quality Bar**|Production|
|**Key Question**|"Is the game healthy and growing?"|

**Entry Criteria:**

- [ ] Global launch complete
- [ ] Live ops team in place
- [ ] Content pipeline running

**Ongoing Success Criteria:**

- [ ] Retention stable or improving
- [ ] Revenue meeting targets
- [ ] Player sentiment positive
- [ ] Content cadence maintained

# 2A.2 Vertical Slice Definition

## Why It Matters

The vertical slice is arguably the most important milestone. It answers the existential question: "Is this game worth making?" A good vertical slice is polished enough to evaluate the experience, representative enough to predict the full game, and small enough to produce quickly. Getting this wrong wastes months building on an unvalidated foundation.

## Framework: Vertical Slice Anatomy

```
                    ┌──────────────────────────────────────┐
                    │         VERTICAL SLICE               │
                    │   (Cross-section of final game)      │
                    └──────────────────────────────────────┘
                                      │
        ┌─────────────────────────────┼─────────────────────────────┐
        │                             │                             │
        ▼                             ▼                             ▼
┌───────────────┐           ┌───────────────┐           ┌───────────────┐
│  CORE LOOP    │           │   CONTENT     │           │   QUALITY     │
│  Complete     │           │   Sample      │           │   Target      │
│               │           │               │           │               │
│ All mechanics │           │ 1-3 levels,   │           │ Art, audio,   │
│ that form the │           │ environments, │           │ UX at final   │
│ minute-to-    │           │ or scenarios  │           │ quality bar   │
│ minute play   │           │ representative│           │               │
└───────────────┘           └───────────────┘           └───────────────┘
```

## Framework: Vertical Slice Scope Definition

### What MUST Be In the Vertical Slice

|Category|Must Include|Example|
|---|---|---|
|**Core Mechanics**|All mechanics the player uses in core loop|Combat, building, matching|
|**Primary Feedback**|Juice, effects, sound for core actions|Hit reactions, collection effects|
|**Progression Hook**|Visible path showing why player continues|XP gain, unlock preview|
|**Representative Content**|Enough to demonstrate variety|3 levels, 5 enemies, 2 environments|
|**Target Quality Art**|At least one area at final quality|Hero character, main environment|
|**Core UI**|Essential interface for playing|HUD, core menus|
|**Audio Foundation**|Music and key sound effects|Main theme, core action sounds|

### What Should NOT Be In the Vertical Slice

|Category|Defer Until Later|Why|
|---|---|---|
|**Meta systems**|Full progression, unlocks, economy|Core loop must be fun without them|
|**Tutorial**|Onboarding, hand-holding|Test if mechanics are intuitive|
|**Social features**|Multiplayer, leaderboards, guilds|Core single-player experience first|
|**Monetization**|Store, IAP, ads|Unless essential to loop|
|**Settings**|Options, accessibility|Polish phase|
|**Full content**|All levels, all characters|Representative sample only|

## Framework: Vertical Slice Quality Targets

|Element|Placeholder OK?|Target Quality|Notes|
|---|---|---|---|
|Core character/unit|No|Final|Player spends most time with this|
|Core environment|No|Final|First impression of world|
|Secondary characters|Yes|Silhouette + animation|Can be refined|
|UI|Partially|Functional, styled|Polish later|
|Audio|Partially|Key sounds final, music WIP||
|VFX|No|Final for core actions|Feel is crucial|
|Performance|No|Target device|Must prove it runs|

|Criterion|Question|Score (1-5)|Evidence|
|---|---|---|---|
|**Core Loop Fun**|Is the minute-to-minute gameplay enjoyable?|||

## Checklist

- [ ] Vertical slice scope defined and documented
- [ ] All core mechanics identified and included
- [ ] Content quantity appropriate (5-15% of final)
- [ ] Quality targets set per element
- [ ] Evaluation criteria agreed upon
- [ ] External playtest plan ready
- [ ] Timeline for vertical slice achievable
- [ ] Go/no-go criteria defined

## Tasks

1. **List every mechanic** in your core loop—each must be in the vertical slice
2. **Define your content sample** specifically (which levels, which characters)
3. **Create a quality checklist** for each vertical slice deliverable
4. **Plan 3+ external playtests** during vertical slice development
5. **Set your minimum score** for proceeding to Alpha

# 2A.3 Soft Launch Success Criteria

## Why It Matters

Soft launch is your reality check. Lab playtests don't predict market performance; only real users with real money in a real store do. Without pre-defined success criteria, teams either rationalize poor data ("the metrics will improve with more content") or panic at normal variance. Set your goalposts before you see the score.

## Framework: Soft Launch Decision Matrix

|Scenario|Data Signal|Decision|Next Steps|
|---|---|---|---|
|**Green Light**|All Tier 1 met, Tier 2 trending positive|Global Launch|Finalize, scale UA|
|**Yellow - Retention**|D1 good, D7/D30 weak|Iterate|FTUE improvements, progression tuning|
|**Yellow - Monetization**|Retention good, ARPU low|Iterate|Economy rebalance, offer optimization|
|**Yellow - Technical**|Metrics good, crashes high|Fix|Stability sprint before global|
|**Red - Core Loop**|D1 <25%, players aren't returning|Major Pivot or Kill|Fundamental redesign needed|
|**Red - No Signal**|All metrics below threshold|Kill|Core experience doesn't resonate|

## Framework: Soft Launch Timeline

|Week|Focus|Key Activities|Decision Point|
|---|---|---|---|
|1-2|Technical Validation|Crash monitoring, performance|Fix critical issues|
|2-4|Retention Analysis|Funnel analysis, drop-off points|Identify FTUE issues|
|4-8|Monetization Testing|Offer testing, price testing|Economy adjustments|
|8-12|Iteration Cycles|Implement fixes, remeasure|Iterate or proceed|
|12-16|Final Validation|Confirm improvements stuck|Go/No-Go Global|

## Framework: Soft Launch Market Selection

|Factor|Criteria|Example Markets|
|---|---|---|
|**Cost**|Low CPI for testing|Philippines, Vietnam, Brazil|
|**Behavior**|Similar to target markets|Australia, Canada, Netherlands|
|**Data Quality**|Reliable analytics, payments work|Nordic countries, Australia|
|**Scale**|Large enough for statistical significance|5,000-10,000 installs minimum|
|**Competition**|Not oversaturated with your genre|Research current top charts|

**Recommended Soft Launch Strategy:**

|Phase|Markets|Purpose|Duration|
|---|---|---|---|
|Phase 1: Technical|Philippines or similar|Crash testing, basic metrics|2-4 weeks|
|Phase 2: Behavioral|Australia, Canada, or similar|Retention, monetization, behavior|4-8 weeks|
|Phase 3: Scale|Add more Tier 1 markets|Validate at scale|2-4 weeks|

## Checklist

- [ ] Retention targets set with specific numbers
- [ ] Monetization targets set (if F2P)
- [ ] Kill thresholds defined
- [ ] Analytics fully instrumented
- [ ] Soft launch markets selected
- [ ] Timeline planned with decision points
- [ ] Team aligned on success criteria
- [ ] Stakeholders signed off on thresholds

## Tasks

1. **Research genre benchmarks** for your specific game type
2. **Set your targets** with minimum, target, and stretch
3. **Define your kill threshold** explicitly—get stakeholder sign-off

# 2A.4 Kill Criteria Definition

## Why It Matters

Kill decisions are emotionally impossible to make in the moment. By then, you've invested months of work, team morale is tied to success, and sunk cost fallacy kicks in. Pre-committing to kill criteria while you're objective is the only way to make rational decisions later. This isn't pessimism—it's discipline.

## Framework: Kill Criteria Categories

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         KILL CRITERIA FRAMEWORK                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   ┌─────────────────┐   ┌─────────────────┐   ┌─────────────────┐      │
│   │   PROTOTYPE     │   │ VERTICAL SLICE  │   │   SOFT LAUNCH   │      │
│   │                 │   │                 │   │                 │      │
│   │ • Technical     │   │ • Fun signal    │   │ • Retention     │      │
│   │   feasibility   │   │ • Player        │   │ • Monetization  │      │

│   │ • Resource      │   │ • Differentiation│  │ • Technical     │      │

│   └─────────────────┘   └─────────────────┘   └─────────────────┘      │

└─────────────────────────────────────────────────────────────────────────┘

## Framework: Kill Criteria by Phase

### Prototype Phase Kill Criteria

|Criterion|Kill If|Rationale|
|---|---|---|

# 2A.5 Feature Prioritization Framework

## Why It Matters

Every project has more ideas than capacity. Without a rigorous prioritization framework, decisions become political (loudest voice wins), arbitrary (most recent idea wins), or paralyzed (nothing gets decided). A structured approach ensures the highest-value work gets done first.

## Framework: MoSCoW Method

|Category|Definition|Guideline|
|---|---|---|
|**Must Have**|Non-negotiable for launch; without these, no shippable product|60% of effort|
|**Should Have**|Important but not critical; workarounds exist|20% of effort|
|**Could Have**|Desirable if time permits; nice polish|15% of effort|
|**Won't Have (This Time)**|Explicitly out of scope; valuable to document|0% of effort|

## Framework: Weighted Scoring Model

For complex prioritization with multiple factors:

|Feature|User Value (×3)|Business Value (×3)|Effort (×2, inverted)|Risk (×1, inverted)|Total|
|---|---|---|---|---|---|
|||||||

**Scoring Guide:**

- User Value: How much does this improve player experience? (1-10)
- Business Value: How much does this improve revenue/retention? (1-10)
- Effort: How much work? (10=easy, 1=massive—inverted so low effort scores high)
- Risk: How risky? (10=safe, 1=very risky—inverted so low risk scores high)

# 2A.7 Estimation Techniques

## Why It Matters

Bad estimates create bad plans. Underestimation leads to crunch and cut features; overestimation wastes resources and delays launches. No estimation method is perfect, but disciplined techniques reduce variance and create honest conversations about uncertainty.

## Framework: Estimation Methods Comparison

|Method|Best For|Accuracy|Effort|
|---|---|---|---|
|**T-Shirt Sizing**|Early planning, rough scoping|Low-Medium|Very Low|
|**Story Points**|Sprint planning, relative sizing|Medium|Low|
|**Reference-Based**|Similar past work|Medium-High|Medium|
|**Three-Point**|Uncertain work|High|Medium|
|**Bottom-Up**|Detailed planning|High|High|

## Framework: T-Shirt Sizing

Quick, relative estimation for high-level planning:

|Size|Effort Equivalent|Risk Level|Examples from Your Project|
|---|---|---|---|
|**XS**|<1 day|Low||
|**S**|1-3 days|Low||
|**M**|1-2 weeks|Medium||
|**L**|2-4 weeks|Medium-High||
|**XL**|1-2 months|High||
|**XXL**|2+ months|Very High|Break this down|

**Rule:** Anything XXL must be broken into smaller pieces before committing.

## Framework: Reference-Based Estimation

Use past experience to calibrate estimates:

|Task Type|Past Example|Actual Time|New Task|Estimated Time|
|---|---|---|---|---|
|Character implementation|Hero character|3 weeks|New character||
|Level design|Tutorial level|2 weeks|Level 2||
|System implementation|Inventory|4 weeks|Crafting||
||||||
||||||

**Calibration Factor:** If past estimates were off, apply a multiplier.

Historical accuracy: Estimates were ___% of actual on average.

Calibration multiplier: 1 / ___% = ___×

## Framework: Three-Point Estimation

For uncertain work, estimate three scenarios:

|Task|Optimistic (O)|Most Likely (M)|Pessimistic (P)|PERT Estimate|
|---|---|---|---|---|
||||||
||||||
||||||
||||||

**PERT Formula:** (O + 4×M + P) / 6 = Expected Duration

**Standard Deviation:** (P - O) / 6 = Uncertainty

**Example:**

- O = 2 weeks, M = 3 weeks, P = 8 weeks
- PERT = (2 + 12 + 8) / 6 = 3.67 weeks
- SD = (8 - 2) / 6 = 1 week
- 95% confidence: 3.67 ± 2 weeks = 1.67 to 5.67 weeks

# 2A.8 Buffer & Contingency Planning

Every plan encounters the unexpected. Without built-in buffer, any variance becomes a crisis. But too much buffer wastes resources. The art is calibrating buffer to risk: more for uncertain work, less for well-understood tasks. Buffer isn't padding—it's risk management.

## Framework: Buffer Types

|Buffer Type|Purpose|Typical Amount|Where Applied|
|---|---|---|---|
|**Task Buffer**|Uncertainty in individual tasks|10-25%|Per task estimate|
|**Integration Buffer**|Combining systems takes time|10-20%|At integration points|
|**Milestone Buffer**|Things go wrong across phase|15-25%|End of each milestone|
|**Project Buffer**|Major unknowns, strategic reserve|10-20%|Project end|
|**Risk Buffer**|Specific identified risks|Risk-specific|When risk occurs|

# 2A.9 Sprint vs. Release Train Cadence

## Why It Matters

Cadence creates rhythm. Without it, work expands to fill available time, priorities constantly shift, and completion feels arbitrary. The right cadence balances iteration speed with meaningful completion. But the right cadence for prototype differs from production differs from live ops.

## Framework: Cadence Options

|Cadence Type|Sprint Length|Best For|Trade-offs|
|---|---|---|---|
|**Weekly Sprints**|1 week|Rapid prototyping, high uncertainty|Overhead, shallow work|
|**2-Week Sprints**|2 weeks|Most development, balanced|Standard choice|
|**3-Week Sprints**|3 weeks|Complex features, senior teams|Harder to adjust|
|**Monthly Cycles**|4 weeks|Content production, live ops|Less agility|
|**Release Trains**|6-12 weeks|Coordinated releases, large teams|Heavy process|

## Framework: Phase-Appropriate Cadence

|Phase|Recommended Cadence|Rationale|
|---|---|---|
|Prototype|1-week sprints|Maximum iteration, fast learning|
|Vertical Slice|2-week sprints|Balance iteration with meaningful work|
|Alpha|2-week sprints|Feature development pace|
|Beta|2-week sprints or weekly|Bug fixing needs quick cycles|
|Soft Launch|2-week sprints|Iteration based on data|
|Live Ops|Monthly release trains|Predictable content cadence|

## Framework: Sprint Structure Template (2-Week)

|Day|Activity|
|---|---|
|**Day 1 (Mon)**|Sprint Planning (2-4 hours): Review backlog, commit to sprint|
|Day 2-4|Development|

|Day 6-9|Development + Integration|
|**Day 10 (Fri)**|Sprint Review (1 hour): Demo completed work|

# 2A.13 Go/No-Go Decision Framework

## Why It Matters

Milestone gates only work if someone actually makes decisions at them. Without a clear framework for go/no-go decisions, milestones become rubber stamps or endless debates. A structured framework ensures decisions are made with the right data, by the right people, at the right time.

## Framework: Decision Gate Structure

```
┌─────────────────────────────────────────────────────────────────────┐
│                      GO/NO-GO DECISION GATE                         │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   ┌─────────────────┐                                               │
│   │  INPUTS         │                                               │
│   │  • Deliverables │                                               │
│   │  • Data/Metrics │                                               │
│   │  • Team assess. │                                               │
│   │  • Risk update  │                                               │
│   └────────┬────────┘                                               │
│            │                                                        │
│            ▼                                                        │
│   ┌─────────────────┐         ┌─────────────────┐                  │
│   │  EVALUATION     │────────▶│   DECISION      │                  │
│   │                 │         │                 │                  │
│   │  Against pre-   │         │  GO / NO-GO /   │                  │
│   │  defined        │         │  CONDITIONAL    │                  │
│   │  criteria       │         │                 │                  │
│   └─────────────────┘         └────────┬────────┘                  │
│                                        │                           │
│            ┌───────────────────────────┼───────────────────────┐   │
│            │                           │                       │   │

│   ┌─────────────┐             ┌─────────────┐         ┌─────────────┐

│   │             │             │             │         │           │ │

# 2A.15 Replanning Protocol

## Why It Matters

No plan survives contact with reality. What matters is how quickly and effectively you replan when circumstances change. A structured replanning protocol prevents both paralysis ("we're off plan but don't know what to do") and chaos ("everyone is constantly changing priorities").

## Framework: Replanning Triggers

|Trigger|Description|Response Level|
|---|---|---|
|**Sprint variance**|Individual tasks over/under|Sprint adjustment|
|**Milestone risk**|Milestone at risk of missing|Milestone replanning|
|**Scope change**|New requirements or cuts|Scope replanning|
|**Resource change**|Team member joins/leaves|Resource replanning|
|**External shift**|Market, competitor, platform change|Strategic replanning|
|**Pivot**|Fundamental direction change|Full replanning|

## Framework: Replanning Response Levels

```
┌─────────────────────────────────────────────────────────────────────┐
│                     REPLANNING RESPONSE LEVELS                      │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  LEVEL 1: Sprint Adjustment                                         │
│  ┌───────────────────────────────────────────────────────────────┐  │
│  │ • Owner: Team lead                                            │  │
│  │ • Timeline: Same day                                          │  │
│  │ • Actions: Reprioritize sprint, adjust assignments            │  │
│  │ • Communication: Team standup                                 │  │
│  └───────────────────────────────────────────────────────────────┘  │
│                                                                     │
│  LEVEL 2: Milestone Replanning                                      │
│  ┌───────────────────────────────────────────────────────────────┐  │
│  │ • Owner: Project lead                                         │  │
│  │ • Timeline: 1-3 days                                          │  │
│  │ • Actions: Resequence work, adjust milestone scope/date       │  │
│  │ • Communication: Stakeholder update                           │  │
│  └───────────────────────────────────────────────────────────────┘  │
│                                                                     │
│  LEVEL 3: Scope Replanning                                          │
│  ┌───────────────────────────────────────────────────────────────┐  │
│  │ • Owner: Product owner + stakeholders                         │  │
│  │ • Timeline: 1 week                                            │  │
│  │ • Actions: Revisit scope boundaries, cut/add features         │  │
│  │ • Communication: Team all-hands, stakeholder meeting          │  │
│  └───────────────────────────────────────────────────────────────┘  │
│                                                                     │
│  LEVEL 4: Strategic Replanning                                      │
│  ┌───────────────────────────────────────────────────────────────┐  │
│  │ • Owner: Leadership + stakeholders                            │  │
│  │ • Timeline: 1-2 weeks                                         │  │
│  │ • Actions: Revisit vision, major scope changes, timeline reset│  │
│  │ • Communication: Full team, external stakeholders             │  │
│  └───────────────────────────────────────────────────────────────┘  │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## Framework: Replanning Process

|Step|Action|Output|
|---|---|---|
|1. Detect|Identify trigger event|Trigger documented|
|2. Assess|Determine response level|Level selected|
|3. Convene|Gather appropriate people|Meeting scheduled|
|4. Analyze|Understand options and trade-offs|Options documented|
|5. Decide|Choose path forward|Decision made|
|6. Update|Revise plans and documents|Updated artifacts|
|7. Communicate|Inform affected parties|Communication sent|
|8. Monitor|Track new plan effectiveness|Ongoing|

## Framework: Trade-off Analysis

When replanning, evaluate options against constraints:

|Option|Scope Impact|Timeline Impact|Quality Impact|Team Impact|Recommendation|
|---|---|---|---|---|---|
|A: Cut feature X|-1 feature|On time|No change|Normal||
|B: Extend timeline|No change|+2 weeks|No change|Normal||
|C: Reduce quality|No change|On time|Lower polish|Normal||
|D: Add resources|No change|Minor delay|No change|Onboarding cost||
|E: Crunch|No change|On time|Risk|Burnout risk||

**Iron Triangle Rule:** You can't change scope, timeline, AND quality simultaneously without trade-offs.
